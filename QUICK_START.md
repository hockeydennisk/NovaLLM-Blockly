# 快速開始指南

## 5分鐘上手 NVT Magic Block

### Step 1: 環境準備

```bash
# 複製環境變數範例
cp .env.example .env
```

編輯 `.env` 文件（如果有 NovaLLM API）：
```
VITE_NOVALLM_API_URL=your_api_url
VITE_NOVALLM_API_KEY=your_api_key
```

**注意**：如果沒有設定 API，系統會自動進入 Demo 模式，仍可正常使用所有功能。

### Step 2: 安裝與啟動

```bash
# 安裝依賴
npm install

# 啟動開發伺服器
npm run dev
```

瀏覽器會自動打開 `http://localhost:5173`

### Step 3: 第一個 Prompt（30秒完成）

#### 情境：創建一個「程式碼審查助手」

1. **拖曳 Persona 積木** 🎭
   - 從左側「AI Prompt Blocks」分類拖曳到工作區
   - 設定角色名稱：`資深程式專家`
   - 設定專長領域：`Code Review`
   - 選擇語氣：`專業嚴謹`

2. **拖曳 Task 積木** 📋
   - 從工具箱拖曳到 Persona 積木下方
   - 點擊 Text 積木並輸入：`請審查以下程式碼並提供改進建議`

3. **拖曳 Context 積木** 📝
   - 連接到 Task 積木下方
   - 準備貼上要審查的程式碼

4. **拖曳 Constraint 積木** ⚠️
   - 勾選「使用條列式」
   - 設定字數限制：300
   - 選擇語言：繁體中文

5. **拖曳 Output Format 積木** 📤
   - 選擇：標題+摘要+建議

**完成！** 右側 Prompt 預覽區會即時顯示組合後的提示詞。

### Step 4: 執行與儲存

1. **執行**：
   - 點擊右上方「執行」按鈕
   - 查看 Result 區域的 AI 回應

2. **儲存模板**：
   - 點擊上方工具列「儲存」
   - 輸入模板名稱：`Code Review 助手`
   - 下次使用時直接「載入」即可

3. **分享給同事**：
   - 點擊「分享」按鈕
   - XML 自動複製到剪貼簿
   - 傳給同事後，他們可以貼上載入

## 6種 AI 積木介紹

### 🎭 Persona（角色積木）

**用途**：定義 AI 的角色與專業

**設定項目**：
- 角色名稱：如「資深工程師」、「財經分析師」
- 專長領域：具體的專業領域
- 語氣風格：專業嚴謹 / 親切友善 / 簡潔直接

**範例**：
```
角色：資深韌體工程師
專長：Display Driver, SoC
語氣：專業嚴謹
```

**生成的 Prompt**：
> 你是資深韌體工程師，擅長 Display Driver, SoC。請使用專業嚴謹的語氣。

---

### 📋 Task（任務積木）

**用途**：清楚描述要完成的任務

**設定項目**：
- 任務輸入：連接文字積木或其他輸入
- 詳細說明：補充任務的細節

**範例**：
```
任務：分析測試 log 並找出根本原因
詳細說明：請特別注意記憶體相關的錯誤
```

**生成的 Prompt**：
> 請協助完成以下任務：
> 分析測試 log 並找出根本原因
>
> 詳細說明：
> 請特別注意記憶體相關的錯誤

---

### 📝 Context（情境積木）

**用途**：提供背景資訊或輸入資料

**設定項目**：
- 情境資料：連接文字輸入

**範例**：
```
情境：[貼上測試 log]
```

**生成的 Prompt**：
> 情境/輸入資料：
> [你的 log 內容]

---

### ⚠️ Constraint（限制條件積木）

**用途**：設定輸出的限制與規範

**設定項目**：
- 使用條列式：勾選後以條列方式輸出
- 不使用行銷語：避免浮誇的形容詞
- 字數限制：設定最大字數（0 表示不限制）
- 語言：繁體中文 / 簡體中文 / English

**範例**：
```
✓ 使用條列式
✓ 不使用行銷語
字數限制：500
語言：繁體中文
```

**生成的 Prompt**：
> 請遵守以下限制條件：
> - 使用條列式呈現
> - 不使用行銷語言
> - 字數不超過 500 字
> - 使用繁體中文

---

### 📤 Output Format（輸出格式積木）

**用途**：指定輸出的格式

**格式選項**：
- 標題+摘要+建議：結構化輸出
- JSON：以 JSON 格式輸出
- Markdown：使用 Markdown 語法
- 表格：表格形式呈現
- 純文字：簡單的文字輸出

**範例**：
```
格式：標題+摘要+建議
```

**生成的 Prompt**：
> 請以下列格式輸出：
> - 標題
> - 重點摘要
> - 行動建議

---

### ✨ Optimizer（Prompt 優化積木）

**用途**：使用 AI 自動優化你的 Prompt

**特殊功能**：
- 雙階段 AI：先優化 Prompt，再執行
- 自動優化：勾選後自動呼叫 NovaLLM 優化

**使用方式**：
1. 將原始 Prompt 連接到 Optimizer 的輸入孔
2. 勾選「自動優化」
3. 系統會先呼叫 AI 優化這個 Prompt
4. 優化後的結果會顯示在 Prompt 預覽區

**價值**：
- 學習如何寫好 Prompt
- 提升 Prompt 品質
- 教育使用者

## 常見使用場景

### 場景 1：QA 工程師分析測試 Log

**積木組合**：
1. Persona: 資深 QA 工程師
2. Task: 分析測試 log 並找出根本原因
3. Context: [貼上 log]
4. Constraint: 條列式 + 繁中 + 300 字
5. Output: 標題+摘要+建議

**適用時機**：每次測試失敗需要分析時

---

### 場景 2：PM 整理會議紀錄

**積木組合**：
1. Persona: 專業會議記錄員
2. Task: 整理會議內容成結構化紀錄
3. Context: [貼上會議逐字稿或錄音轉文字]
4. Constraint: 條列式 + 繁中
5. Output: Markdown

**適用時機**：每次會議結束後

---

### 場景 3：工程師 Code Review

**積木組合**：
1. Persona: 資深程式專家
2. Task: 審查程式碼並提供改進建議
3. Context: [貼上程式碼]
4. Constraint: 條列式 + 不用行銷語 + 繁中
5. Output: 標題+摘要+建議

**適用時機**：Pull Request 審查時

---

### 場景 4：ADM 產生 Email 草稿

**積木組合**：
1. Persona: 專業商務溝通專家
2. Task: 根據以下要點產生 Email 草稿
3. Context: [輸入要點]
4. Constraint: 親切友善 + 繁中 + 200 字
5. Output: 純文字

**適用時機**：需要撰寫正式 Email 時

## 進階技巧

### 技巧 1：使用變數

1. 從「Variables」分類創建新變數
2. 在積木中引用變數
3. 適合需要重複使用的資料

**範例**：
```
變數「專案名稱」= "Display Driver"
Task: 分析 {專案名稱} 的測試結果
```

### 技巧 2：組合 Logic 積木

使用 Logic 分類的積木增加條件判斷：

```
IF 測試失敗
  THEN Persona: QA 工程師 → 分析原因
  ELSE Persona: PM → 產生成功報告
```

### 技巧 3：建立模板庫

**建議分類**：
- 📁 程式開發
  - Code Review
  - Bug 分析
  - API 設計

- 📁 專案管理
  - 會議紀錄
  - 進度報告
  - 風險評估

- 📁 測試 QA
  - Log 分析
  - 測試計畫
  - 缺陷報告

- 📁 文件撰寫
  - Email 草稿
  - 技術文件
  - 使用手冊

## 疑難排解

### Q: Prompt 沒有即時更新？

**A**: 確認積木有正確連接，檢查是否有紅色警告標記。

### Q: 執行沒有反應？

**A**:
1. 檢查 `.env` 是否設定（Demo 模式仍可用）
2. 確認 Prompt 區域有內容
3. 查看瀏覽器 Console 是否有錯誤

### Q: 如何匯入同事分享的模板？

**A**:
1. 複製同事提供的 XML
2. 點擊「載入」→ 選擇「貼上 XML」
3. 工作區會自動載入積木

### Q: 積木太多找不到怎麼辦？

**A**:
使用左側工具箱的分類：
- AI Prompt Blocks：AI 專用積木
- Logic：條件判斷
- Text：文字處理
- Variables：變數管理
