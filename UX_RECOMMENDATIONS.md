## 關鍵問題解答

### Q1: 任務串連與資料傳遞

**你的問題**：
> 如何讓前一個任務的結果傳入下一個任務？

**答案：完全可行且必要！**

#### 設計方案

##### 方案 A：Pipeline 模式（推薦）

```
[任務A: 分析資料] ─輸出─→ [任務B: 產生報告] ─輸出─→ [任務C: 翻譯]
```

**實現方式**：
1. 每個任務積木有「輸出孔」（右側）
2. 下一個任務積木有「輸入孔」（左側）
3. 連接時可選擇要用哪部分資料：
   - 完整回覆
   - 摘要
   - JSON 特定欄位

**UX 呈現**：
```
┌──────────────────────┐
│ 📋 任務A: 分析 log   │
│                      │◉ 輸出
└──────────────────────┘
        │
        ↓
┌──────────────────────┐
◉ 使用前任務結果       │
│ 📋 任務B: 產生報告   │
│                      │◉ 輸出
└──────────────────────┘
```

**程式實現**（已部分完成）：
- 使用變數系統儲存前一任務結果
- 在生成 Prompt 時替換變數
- 串連執行時依序呼叫 API

##### 方案 B：變數引用模式（輔助）

```
[執行任務A] → 儲存到變數「分析結果」
[任務B] → 引用變數「分析結果」
```

**適用情境**：
- 不一定按照順序執行
- 多個任務引用同一結果
- 需要手動控制執行流程

#### 可行性分析

| 項目 | 難度 | 時間 | 價值 |
|------|------|------|------|
| 基礎串連 | ⭐⭐ | 1天 | ⭐⭐⭐⭐⭐ |
| 變數系統 | ⭐⭐⭐ | 2天 | ⭐⭐⭐⭐ |
| 資料選擇 | ⭐⭐ | 1天 | ⭐⭐⭐⭐ |

**競賽建議**：
- MVP：基礎串連即可（順序執行）
- 進階：加入變數引用

### Q2: Scratch 的「角色」概念適合嗎？

**你的疑問**：
> Scratch 的角色概念是否適用？

**答案：適合，但需要「語意升級」！**

#### Scratch vs. 我們的差異

| 面向 | Scratch | NVT Magic Block |
|------|---------|-----------------|
| 角色定義 | 視覺角色（貓、狗） | AI 人格（專家、分析師） |
| 角色行為 | 移動、說話、造型 | 思考方式、專業領域、語氣 |
| 使用場景 | 教育、遊戲 | 企業工作流程 |
| 角色切換 | 舞台上多個角色 | 同一流程中切換 Persona |

#### 我們的 Persona 系統設計

**基礎版（已實現）**：
```
🎭 Persona
   - 角色名稱：資深程式專家
   - 專長領域：React、TypeScript
   - 語氣：專業嚴謹
```

**進階版（建議）**：

##### 1. 全域 Persona vs. 區域 Persona

```
🌍 全域 Persona: 資深工程師
   ├─ 📋 任務A: 分析程式碼
   ├─ 📋 任務B: 提出改進建議
   └─ 🎭 切換 Persona: 專案經理
       └─ 📋 任務C: 產生管理報告
```

**UX 呈現**：
- 全域 Persona：影響所有任務
- 區域 Persona：覆蓋特定任務
- 視覺提示：不同顏色或邊框

##### 2. 多角色協作

```
[QA 工程師] → 找出 bug
    ↓
[資深工程師] → 建議修復方案
    ↓
[專案經理] → 評估優先級
```

**應用場景**：
- Code Review：工程師分析 → PM 評估影響
- 測試分析：QA 找問題 → 工程師提方案
- 需求分析：PM 整理需求 → UX 提建議

#### 實現建議

**MVP 階段（已完成）**：
- 單一 Persona 積木
- 設定角色、專長、語氣

**V2 階段**：
- Persona 作用域（全域/區域）
- Persona 切換積木
- 視覺化顯示當前生效的 Persona

**V3 階段**：
- Persona 模板庫
- 自定義 Persona 積木
- Persona 權限管理（部門專屬）

## 進階 UX 優化建議

### 1. Prompt 組成視覺化

**問題**：使用者不知道最終 Prompt 由哪些積木組成

**解決方案**：
在 Prompt 預覽區加入「來源標記」

```
🎭 [Persona] 你是資深程式專家，擅長 React
📋 [Task] 請協助審查以下程式碼
⚠️ [Constraint] 請使用條列式，不超過 300 字
📤 [Output] 請以 Markdown 格式輸出
```

**好處**：
- 教育使用者
- 易於除錯
- 展示 Prompt 結構

### 2. 智慧建議系統

**情境**：使用者拖了 Task 積木但沒加 Constraint

**系統建議**：
```
💡 建議加入以下積木：
   - Constraint（避免回覆過長）
   - Output Format（結構化輸出）
```

**實現**：
- 分析工作區積木
- 根據最佳實踐提示
- 一鍵加入建議積木

### 3. Prompt 品質評分

**功能**：即時評估 Prompt 品質

```
Prompt 品質：⭐⭐⭐⭐ (80/100)
✅ 有明確角色
✅ 有具體任務
✅ 有限制條件
⚠️ 建議加入輸出格式
```

**評分維度**：
- 是否有 Persona
- Task 是否明確
- 是否有 Constraint
- 是否有 Output Format

### 4. 熱力圖與成效追蹤

**功能**：顯示積木使用熱度

```
📊 積木使用統計
1. Persona: 資深程式專家 (使用 128 次，好評 96%)
2. Task: Code Review (使用 89 次，好評 92%)
3. Constraint: 條列式+繁中 (使用 156 次)
```

**價值**：
- 發現熱門組合
- 優化模板庫
- 企業知識萃取

**旁白**：
> 每次用 AI，都要重新想一遍 Prompt...

#### 第二幕：解決方案（我們的方式）

**畫面**：NVT Magic Block 介面

```
┌───────────┬─────────────┐
│ 拖曳 5 個 │ 即時生成    │
│ 積木      │ Prompt      │
│           │             │
│ (動畫展示)│ (逐步顯示) │
└───────────┴─────────────┘
```

**旁白**：
> 拖曳積木，組合 Prompt
> 儲存模板，隨時重用
